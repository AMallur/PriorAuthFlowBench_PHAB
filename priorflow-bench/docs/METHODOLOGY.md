# Methodology

This follows the structure of the MedAgentBench methodology (Jiang et al., NEJM AI) as closely as is honest to do solo, and explicitly marks where the two diverge.

## 1. Motivation

Existing agent benchmarks for healthcare (MedAgentBench included) evaluate general clinical EHR interaction — retrieving labs, placing orders, checking vitals. None evaluate the specific reasoning pattern required for **payer-facing prior authorization**: reading a chart against externally-imposed, often non-obvious criteria (step therapy exceptions, setting-dependent exemptions, documentation staleness windows) and producing a decision or artifact (an auth determination or an appeal letter) that must be defensible if audited. This is a distinct skill from general EHR question-answering, and it's the skill oncology RCM analysts are actually hired for.

## 2. Environment construction

- **Server:** HAPI FHIR JPA server (R4), the standard open-source reference implementation, run via the included `docker-compose.yml`.
- **Patient population:** two separate pools, not one. (1) A bulk, unlabeled population generated via Synthea (`scripts/generate_synthetic_patients.sh`), biased toward oncology modules (breast, colorectal, lung cancer) — meant as general-purpose filler/noise data, not tied to any specific task. (2) 22 hand-crafted patients, one per seed task, generated deterministically from `synthetic_data/patient_specs.py` via `synthetic_data/generate_bundles.py` — these exist specifically so each task's `required_fhir_resources` and `reference_solution` are actually backed by real FHIR resources in the loaded server, not just asserted in `task_bank.json`. Load them with `scripts/load_task_bank_patients.sh`.
- **Relative-date facts:** several tasks hinge on a time window relative to "now" (e.g. pab_005's labs being 41 days old, pab_009's ECOG from "last week", pab_011's discharge "yesterday"). `generate_bundles.py` computes these dates relative to its run time, not fixed calendar dates — regenerate (the load script does this automatically) shortly before running an eval, or these facts will drift as real time passes.
- **Divergence from MedAgentBench:** MedAgentBench drew 100 patients from Stanford's real (de-identified) STARR warehouse via institutional data access. We do not have that access, and would not seek it for a project at this stage even if we could — Synthea data carries no PHI risk, which is the correct tradeoff for a benchmark intended to be published openly. The tradeoff is that synthetic charts are less messy/realistic than real ones; this should be stated as a limitation, not hidden.

## 3. Task design

- **Author:** currently one person (RCM/oncology billing background, ContractorCloud/PriorFlow founder), analogous to the domain-expert role physicians played for MedAgentBench.
- **Divergence:** MedAgentBench tasks were written/reviewed by multiple physicians, giving some inter-rater validation on task correctness. A solo-authored task bank has no such check. Until a second domain reviewer is brought in, tasks should be labeled `reviewed_by: 1` in metadata and the limitation stated plainly in any public write-up.
- **Design principle used:** paired positive/negative tasks per hard case (see `tasks/categories.md`), to reduce the chance an agent scores well by pattern-matching category → expected answer shape rather than actually reasoning from the chart.
- **`genomic_biomarker_necessity` sourcing:** unlike the other categories, this category's reference solutions must be traceable to real FDA approval records and pivotal trial data (not the author's domain memory alone), because inventing a biomarker-therapy link would make the benchmark actively misleading rather than merely synthetic. Tasks pab_013–pab_022 were drafted against a literature/web-grounded research pass (NTRK fusion → larotrectinib/entrectinib; MSI-H/dMMR → pembrolizumab; KRAS G12C → sotorasib/adagrasib; EGFR exon 20 insertion vs. exon 19 del/L858R → amivantamab vs. osimertinib; germline BRCA1/2 pathogenic variant vs. VUS → olaparib), citing FDA approval dates and pivotal trials (KEYNOTE-158, CodeBreaK 100/200, CHRYSALIS, SOLO-1, POLO). This still needs the same second-reviewer sign-off as every other category before being treated as final — literature grounding reduces the risk of *fabricated* biology, it does not substitute for domain review of the *payer-policy* framing.
- **Stale-knowledge traps:** this category deliberately includes facts with a specific "as-of" date (e.g. mobocertinib's FDA approval followed by its voluntary withdrawal in July 2024; sotorasib's accelerated approval remaining un-converted to full approval after an adverse Oct 2023 ODAC vote on confirmatory data). An agent whose training data predates these events will confidently give an outdated answer. Reference solutions should record the date the underlying fact was last verified, since these facts can change again.

## 4. Harness

- Custom lightweight controller (`harness/controller.py`) rather than reusing AgentBench, because AgentBench's abstractions are built around a broader set of environments (web browsing, OS interaction, etc.) that add complexity not needed here. This is a reasonable engineering simplification, not a corner cut on rigor.
- Scoring blends (a) tool-call trace match — did the agent query the FHIR resources a correct trajectory requires — and (b) final-answer correctness, so an agent can't score well by guessing without looking at the chart.
- **Answer scoring (`harness/scorer.py`):** `exact_match` tasks are graded by deterministic structural equality. `checklist`/`llm_rubric` tasks are graded by an LLM judge (`llm_judge_checklist`) that scores each checklist item against the reference solution via a forced tool call, returning a per-item verdict with a reason — this replaced an earlier keyword-match placeholder that could be satisfied by an answer merely repeating checklist nouns without getting the determination right. The judge client is injectable (`score_task(..., judge_client=...)`) specifically so `tests/test_scorer.py` can verify the grading logic with a fake client, without needing network access or an API key.
- **Test coverage:** `tests/test_scorer.py` covers the scorer/judge wiring with a fake Anthropic client; `tests/test_patient_data_consistency.py` cross-checks every seed task's `required_fhir_resources` and key clinical facts (e.g. "NTRK3 fusion" present for pab_013, absent for pab_014) against the generated patient bundles, so a future edit to either `task_bank.json` or `patient_specs.py` that breaks their correspondence fails a test instead of silently shipping.

## 5. Known limitations (state these explicitly, don't bury them)

1. Solo-authored task bank — no inter-rater task validation yet.
2. Synthetic patient data — less noisy/realistic than a real clinical warehouse. It has also never been loaded into a live HAPI server end-to-end in this environment (no Docker daemon available where this was developed) — the bundles are generated and unit-tested for internal consistency against `task_bank.json`, but a live agent-vs-FHIR-server run has not yet been executed to confirm the harness works end-to-end.
3. Payer policy text is illustrative/synthetic, not sourced from verified current real payer policy documents — do not cite specific numeric thresholds in this repo as representing any real payer's actual current criteria.
4. The LLM-judge scorer (`llm_judge_checklist` in `scorer.py`) has unit tests against a fake client but has not been validated against a human-graded sample — it could still systematically over- or under-grade in ways only a human comparison would reveal. Treat its scores as provisional until that validation happens.
5. Small task count (22 seed tasks vs. MedAgentBench's 300) — treat any score on this version as a smoke test of the harness, not a claim about agent capability at large.
6. `genomic_biomarker_necessity` facts (pab_013–pab_022) are time-sensitive by nature (drug approvals, withdrawals, and label changes) and were last verified via a literature/web research pass in July 2026 — re-verify before citing in any published version, since regulatory status can change.

## 6. Suggested next steps toward a defensible v1

1. Get a second RCM/oncology billing professional (or a contracted oncologist) to independently review and score-check each task's reference solution.
2. Run the harness end-to-end against a live HAPI server (`docker compose up -d`, `scripts/load_task_bank_patients.sh`, `python harness/controller.py`) to confirm the agent → FHIR → scorer pipeline actually works, then validate the LLM-judge's grades against a human-graded sample.
3. Expand to 50-100 tasks once the above two are in place, maintaining the paired positive/negative discipline.
4. Run the reference Claude agent and your production PriorFlow agent side by side, publish both scores — a benchmark that only ever reports the author's own product's score is not credible as a benchmark.
5. Consider a short arXiv writeup once task count and review are solid — arXiv doesn't require institutional affiliation, and a well-executed niche benchmark is a legitimate research contribution and credibility asset independent of PriorFlow's commercial success.
